# 어디에 탈까 — 지하철 1~8호선 자리 잡기 도우미

서울 지하철 1~8호선에서 **어느 칸, 어느 문 앞에 타야 빨리, 그리고 오래 앉아갈 수 있는지** 추천해주는 모바일 웹앱.

출발역·도착역(1~8호선 전체에서 검색, 노선은 자동 선택)과 출발 시각("지금" 또는 시:분 직접
선택 — 통계가 시간 단위라 계산은 해당 시간 버킷 기준)을 고르면, 중간 역들의 하차 통계와
하차 승객이 몰리는 칸 위치(빠른환승 데이터)를 조합해 문 위치별(편성 6~10량 × 4문)로
**앉을 확률**과 **기대 착석 정거장 수**를 계산한다.
출근길(집앞역↔회사역)을 저장해 원탭으로 불러올 수 있고, **출발 시간별 착석 전망 차트**로
언제 출발하는 게 가장 여유로운지 비교할 수 있다.

**환승 1회 여정까지 지원한다**: 직행이 없으면 출발 노선×도착 노선의 공유 역 중
`정거장 수 합 + 환승 페널티(3정거장 상당)`가 최소인 환승역을 골라 구간별로 착석 위치를
각각 추천하고, 환승역에서 **환승 통로와 가까운 문 위치**(빠른환승 데이터)를 부가 정보로
알려준다 — 주 추천은 어디까지나 착석 기준이고 빠른환승은 서브다. 직행과 환승이 모두
가능하면 비용이 낮은 쪽을 자동 선택한다.

노선 형태도 실제 운행대로 처리한다: 2호선 본선 **순환**(내선/외선 짧은 쪽 자동),
6호선 **응암 단방향 루프**(루프 구간은 실운행 순서로 전개), 5호선 **마천지선**(본선/마천행
자동 선택, 지선을 가로지르는 조합은 강동 환승 여정으로 처리). 노선 간 역명이 다른 물리적
동일 역(이수/총신대입구)은 하나로 합쳐 검색·환승에 쓴다.

## 작동 원리 (확률 모델)

문 위치 `i`(8량 × 4문 = 32개) 앞에 서 있을 때, 중간 역 s에서 좌석을 확보할 확률:

```
p_i(s) = 1 − exp(−freed_i(s) / C(s))
freed_i(s) = [alight(s,h) × 방향비율(s,h) / 시간당 열차 수] × W_s[i]
```

- `alight(s,h)` — 역 s의 시(hour)별 일평균 하차 인원 (2025년 실측 통계, 평일/주말 분리).
  열차 1대당 인원으로 환산 (RH 13대/h, 평시 8대/h 근사)
- `방향비율(s,h)` — 하차 통계는 양방향 합산이라, 역·방향·요일·시간별 **열차 혼잡도**
  (지하철혼잡도정보, 정원 대비 %)에 비례해 내 방향 몫을 배분한다
  (내 방향 혼잡도 / 양방향 합, 0.15~0.85 클램프). 혼잡도 없는 역은 50% 폴백
- `W_s[i]` — 역 s 하차 승객이 문 i 근처에 있을 확률: 빠른환승 문 위치 중심
  가우시안 커널(σ=2문)과 균등 분포의 혼합(λ=0.6). 데이터 없는 역은 균등 폴백
- `C(s)` — 좌석 경쟁 상수: 좌석이 빌 때 그 문 구역의 **입석 인원**이 경쟁자.
  혼잡도로 추정해 `C = 1 + κ·max(0, 정원×혼잡도 − 좌석수)/문수` (κ=0.5,
  정원 160명/칸, 좌석 54석/칸). 혼잡도 없는 역은 고정값 3 폴백

출발역 탑승 시점도 모델링한다 — **바로 앉을 확률 p₀**:

```
p₀_i = 1 − exp(−빈자리_i / (1 + κb·경쟁자_i))
빈자리_i  = max(0, 좌석 − 재차 인원)/문수 + 출발역 하차 중 착석자 몫 × W[i]
경쟁자_i  = 승차 인원 × W[i] + 차내 입석/문수
```

재차 인원은 혼잡도로 추정하고(없으면 p₀=0 폴백), 하차로 비는 자리는 차내 입석
승객이 우선하므로 입석을 경쟁자에 넣는다. 승차 인원도 같은 승하차 통계(승차 행)에서
오며, 승차객도 환승 통로 근처 문에 몰린다고 보고 같은 W를 쓴다. κb(기본 1)는
"바로 앉음" 피드백으로 보정할 파라미터다.

p₀에서 시작해, 아직 못 앉았을 확률(생존)을 역 순서대로 곱해가며 두 지표를 얻는다:

```
앉을 확률      = p₀ + Σ_s 생존(s)·p_i(s)
기대 착석 정거장 = p₀·총구간 + Σ_s 생존(s)·p_i(s)·(도착까지 남은 정거장 수)
```

추천 순위는 두 지표의 α 가중 결합: `rank = α·앉을확률 + (1−α)·기대착석/최대착석`.
두 지표는 리스크 성향이 다르다 — 앉을 확률은 도착 직전에라도 **한 번이라도 앉을 확실성**,
기대 착석은 못 앉을 위험을 감수하더라도 **일찍 앉아 오래 가는 기댓값**을 본다.
**추천 성향 슬라이더**로 α를 조절할 수 있다 (0 = 앉는 시간 최대로, 1 = 한 번이라도
확실히, 기본 0.3, localStorage에 저장). 1위와 점수 차가 2% 이내인 다른 칸은
**사실상 동점**으로 함께 보여준다("5-1문도 사실상 동점이에요") — 통계 추정 오차보다
작은 차이로 한 칸만 고집하는 건 무의미해서, 승강장에서 가까운 쪽을 고르면 된다.
추천 하단의 **피드백 위젯**(바로 앉음/가다가
앉음/못 앉음)은 예측과 실제 결과를 localStorage에 쌓아, 추후 경쟁 상수 C·κ를 실측으로
보정할 근거가 된다. "바로 앉음"은 p₀ 예측(predictedBoardProb, κb 보정용)과 대응된다.
기록 항목(v2, `src/core/feedback.ts`): 예측 확률·기대 착석·바로 앉을 확률·당시 α·
여정 길이에 더해, **추천 칸에 실제로 탔는지**(기본 체크된 체크박스 — 아니면 비교에서 제외)와
"가다가 앉음"일 때 **어디서 앉았는지**(선택식 후속 질문, 건너뛰기 가능)를 담는다.
환승 여정도 구간별로 기록되며, **내보내기** 버튼으로 공유 시트(폴백: 클립보드)를 통해
JSON을 백업할 수 있다 — localStorage는 브라우저 데이터 정리로 사라질 수 있어서다.

## 실행·배포

```bash
npm install
npm run dev        # 개발 서버
npm test           # 단위 테스트 (core는 전부 순수 함수)
npm run build      # 정적 빌드 (dist/, base=/shortcut/)
```

UI 스모크 테스트(스크린샷 포함): `npm run preview` 실행 후
`node scripts/ui-smoke.mjs <스크린샷 저장 경로>`

**배포**: https://woosukqw12.github.io/shortcut/ — `main` 푸시마다 GitHub Actions가
테스트→빌드→GitHub Pages 배포(`.github/workflows/deploy.yml`). **PWA**라 폰 브라우저에서
"홈 화면에 추가"하면 앱처럼 열리고, 서비스워커가 방문한 화면·노선 데이터를 캐시해
지하철 안 오프라인에서도 동작한다 (HTML은 network-first라 새 배포가 우선).
아이콘 재생성: `node scripts/gen-icons.mjs`

## 데이터 파이프라인

`src/data/line*.json`(커밋됨)이 앱의 유일한 데이터 소스다. 갱신하려면:

```bash
npx tsx data-pipeline/fetch.ts   # 원천 CSV 다운로드 → data-pipeline/raw/
npm run data:build               # 파싱·검증 후 line7.json 재생성
```

| 원천 | 출처 | 용도 |
|---|---|---|
| 역별 일별 시간대별 승하차인원 (OA-12921) | [서울열린데이터광장](https://data.seoul.go.kr/dataList/OA-12921/F/1/datasetView.do) | 시간대별 일평균 하차 인원 |
| 수도권 도시철도 환승 데이터 (OA-22521) | [서울열린데이터광장](https://data.seoul.go.kr/dataList/OA-22521/F/1/datasetView.do) | 빠른환승 하차 칸/문 위치 |
| 지하철혼잡도정보 (OA-12928) | [서울열린데이터광장](https://data.seoul.go.kr/dataList/OA-12928/F/1/datasetView.do) | 방향 비율·경쟁 상수 보정 (30분 단위·상하선·요일별) |

원천 CSV는 CP949 인코딩이다. **역 목록은 손으로 쓰지 않는다** — 승하차 CSV의 역번호가
노선 순서를 담고 있어 자동 생성하고, 예외(코드 순서가 어긋난 동묘앞·남위례, 게이트를 공유해
통계가 타 노선으로 잡히는 충무로·연신내, 지선 분기, 타 운영사 구간 등)만
`data-pipeline/line-defs.ts`에 기술한다. 역명 표기 차이(`이수`/`총신대입구` 등)는 괄호 제거 +
별칭으로 정규화한다. 포털 다운로드가 실패하면 위 링크에서 수동 다운로드해
`data-pipeline/raw/`에 같은 이름으로 두면 된다.

## 알아둘 한계

1. **하차 인원에는 입석 승객도 포함** — 좌석 회전율의 프록시일 뿐, 앉아 있던 사람 수가 아니다.
2. **탑승 경쟁은 출발역만 반영(근사)** — 출발역 경쟁은 p₀로 모델링했지만, 재차 인원을
   출발역 혼잡도로 근사하고 중간 역에서 새로 타는 승객은 여전히 경쟁 상수 C가
   뭉뚱그려 흡수한다.
3. **하차 통계 자체는 방향 합산** — 혼잡도 데이터로 방향·요일을 보정하지만,
   원천 하차 인원은 상·하행 합산이라 배분은 추정이다. 시간당 열차 수도 근사값이며,
   혼잡도가 없는 역(타 운영사 구간 등)은 50%·C=3 폴백을 쓴다.
4. **서울교통공사 관할 구간만 통계가 있다** — 1호선은 서울역~청량리, 3호선은 지축 이남,
   4호선은 불암산~남태령만 다룬다. 7호선 인천 구간(까치울~석남)과 8호선 별내선 구간은
   역 선택은 되지만 하차 통계가 없어 계산에서 제외된다 (UI에 표기됨).
5. 2호선 성수지선·신정지선은 미지원. 환승 위치 데이터가 없는 역은 균등 분포로 처리된다.
6. **환승은 1회까지** — 2회 이상 환승이 필요한 조합(예: 1호선↔8호선)은 안내 문구만 띄운다.
   환승 페널티(3정거장 상당)와 두 구간에 같은 출발 시각을 쓰는 것 모두 근사이며,
   1~8호선 밖 노선(9호선·수인분당선 등)으로의 환승 경로는 탐색하지 않는다.
   환승 여정은 출발 시간 차트·피드백 기록을 아직 지원하지 않는다.

## 구조

```
data-pipeline/   # 오프라인 파이프라인: line-defs(노선 정의) → assemble → doors → build
src/data/        # line1~8(+5m).json, lines.json, stations-index.json + zod 스키마
src/core/        # 순수 함수: 경로 전략(linear/circular/loop-tail), 환승 여정 계획, 점수 모델
src/components/  # 모바일 우선 UI (노선 데이터는 여정에 필요할 때 지연 로드)
tests/           # Vitest — core 로직 + 전 노선 실데이터 검증 + 스냅샷
```
